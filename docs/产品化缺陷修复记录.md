# 产品化缺陷修复记录（逻辑/算法漏洞）

本轮针对"从原型走向产品"排查并修复了系统中的逻辑与算法漏洞。所有修复均有离线可复现的回归测试：
`python tests/test_bugfixes.py`（cwd = `python/`）。

---

## 1. 相机实测帧率未参与信号处理（算法·系统性偏差）

- **现象/根因**：主循环用固定的 `cfg["camera_fps"]=30` 驱动所有 rPPG DSP（缓冲长度、FFT→BPM 换算），但从未使用相机实际帧率。任何非 30fps 的相机（如 25/15fps）都会导致心率出现**系统性偏差**（例如实际 25fps 却按 30fps 换算，心率整体偏高约 20%）。
- **修复**：`main.py` 改用相机实测帧率 `camera.fps` 驱动 DSP，并钳制到合理区间 `[10, 60]`，与标称值不一致时打印提示。
- **文件**：[main.py](../python/main.py)

## 2. 滑动缓冲无时序连续性（算法·错误心率）

- **现象/根因**：`SignalBuffer` 是定长环形队列，仅按数量淘汰、不按时间。人脸一旦丢失后又重新捕获，队列里会**混入丢失前后不连续的帧**，对其做 FFT/峰值检测会得到错误心率与 HRV。
- **修复**：`main.py` 记录上次成功入缓冲时间，若间隔超过 `buffer_gap_reset_sec`（默认 1.0s）即判定帧序列断裂，清空 `sig_buf`/`ref_buf`、复位 `IBIAccumulator`（新增 `reset()`）与心率/HRV 状态，待重新积累完整窗口后再计算。
- **文件**：[main.py](../python/main.py)、[rppg/hrv_analyzer.py](../python/rppg/hrv_analyzer.py)

## 3. 自动恢复会在驾驶员仍疲劳时静默报警（逻辑·安全隐患）⚠️

- **现象/根因**：分类器的"自动恢复"机制在任一疲劳等级持续固定时长（如重度 15s）后**无条件清零到 Level 0**（绿灯、静音）。这意味着一个**真正处于重度疲劳的驾驶员，报警会在 15 秒后自动关闭**——安全隐患。该机制原是为演示时"防止长时间响铃"而加。
- **修复**：为自动恢复增加**证据约束**——仅当当前综合评分已低于该等级的降级阈值（证据确实减弱）时才允许超时恢复；若评分仍高，报警必须持续。既保留了对"误报卡死"的自愈能力，又杜绝了对真实疲劳的误静默。
- **文件**：[fusion/fatigue_classifier.py](../python/fusion/fatigue_classifier.py)

## 4. NIR trend 模式的 HRV 计算器每帧重建，趋势判据永久失效（算法·功能性 Bug）

- **现象/根因**：`normalize_hrv_score` 每次调用都 `NIRHRVRiskCalculator(config)` 新建实例，而 `trend` 模式依赖实例内部的历史队列累积斜率。新建即清空历史 → 历史恒为空 → 趋势风险恒为 0，`nir_hrv_mode="trend"` 完全不工作。
- **修复**：`FeatureFuser` 持有**单一** `NIRHRVRiskCalculator` 实例并复用，历史得以跨帧累积。回归测试验证：持续下降的 HRV 在累积足够历史后风险由 0 上升到 0.8。
- **文件**：[fusion/feature_fusion.py](../python/fusion/feature_fusion.py)

## 5. 融合/分级/串口在回退路径按帧率（~30Hz）执行（逻辑·不一致 + 资源浪费）

- **现象/根因**：仅当 rPPG 就绪（约 1Hz）才走"正式"评估，其余每一帧都走回退分支再次执行 `fuse()`+`classify()`+`serial.send()`。这使得 EMA 平滑的时间常数随调用频率漂移（30Hz 下几乎不平滑），并且以帧率刷串口、增加 ARM CPU 负担。
- **修复**：将融合/分级/串口下发重构为**单一固定节律评估块**（`eval_interval_sec` 默认 0.2s=5Hz），无论 rPPG 是否就绪都以恒定节律评估。rPPG 分支只负责更新 `bvp/hr/hrv/sqi`。EMA 行为稳定、串口流量可控。
- **文件**：[main.py](../python/main.py)

## 6. 头姿 solvePnP 每帧重算（性能·ARM 负担）

- **现象/根因**：MediaPipe 每 `mediapipe_interval` 帧才推理一次，中间帧复用缓存 landmark；但头姿 `solvePnP` 仍每帧执行，对同一份 landmark 反复求解，浪费算力。
- **修复**：仅在有新 landmark 时重算头姿，其余帧复用缓存值。PERCLOS/哈欠仍逐帧统计以保持窗口时间语义。
- **文件**：[main.py](../python/main.py)

## 7. 旧兼容接口在 NIR 模式误走 RGB HRV 公式（潜在 Bug）

- **现象/根因**：`fuse_features()` 调用 `normalize_hrv_score(hrv_features)` 未传 config，NIR 模式下也会走 RGB 完整 HRV 公式。
- **修复**：传入 config 以正确路由。
- **文件**：[fusion/feature_fusion.py](../python/fusion/feature_fusion.py)

## 新增/变更配置项（`config.py`）

| 键 | 默认 | 说明 |
|----|------|------|
| `eval_interval_sec` | 0.2 | 融合/分级/串口统一评估节律（秒） |
| `buffer_gap_reset_sec` | 1.0 | ROI 入缓冲间隔超过此值判定帧序列断裂并清空缓冲 |

## 后续建议（未在本轮改动，供产品化排期）

- `_compute_lf_hf`/`compute_bvp_sqi`/`nir_illum` 使用 `np.trapz`，NumPy≥2.0 已弃用（改 `np.trapezoid`），未来大版本或移除，建议加兼容封装。
- 长时间无人脸（驾驶员长时间侧脸/离开）目前依赖行为通道置信度降权，可考虑显式"不可用/未知"状态与提示。
- `FaceTracker` 长时间纯光流可能漂移，现依赖每秒重检测纠偏；产品化可加跟踪置信度回退触发即时重检测。
